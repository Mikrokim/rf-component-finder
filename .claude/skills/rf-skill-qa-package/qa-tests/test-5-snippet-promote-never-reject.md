# מבחן 5 — snippet מקדם, לא פוסל (בדיקת אי־פספוס)

**מטרה:** לוודא שהסקיל לא פוסל רכיב על סמך תוצאת חיפוש/snippet בלבד, אלא רק מול ערך מ־datasheet — כדי שלא נפספס התאמה קיימת רק כי ה־snippet שלה היה חלקי או מטעה.

**רקע:** snippets של דיסטריביוטרים מציגים לעיתים קרובות ערך typ בנקודת תדר אחת, משמיטים תנאים, או פשוט שגויים. אם הסקיל פוסל על סמך snippet, הוא עלול לזרוק חלק טוב לפני שבכלל פתח לו datasheet. זו הבדיקה ההפוכה למבחן 1: שם ודאנו שלא מוצגת התאמת־שווא; כאן מוודאים שלא נזרקת התאמה אמיתית בזול.

**הוראות:** להריץ בשיחה חדשה ונקייה. לוודא שהסקיל rf-component-search הופעל. הפרומפט בוחר מפרט שבו סביר שחלק מהמועמדים יראו ב־snippet ערך גבולי/חלקי (במיוחד NF ו־OP1dB), כדי לבחון אם הם נפסלים מוקדם מדי.

## הפרומפט (להעתקה)

```
מחפשת driver amplifier 12-16GHz, gain 18-22db, OP1dB 22dbm min, NF 3.5db max, כל צורת רכיב
```

## קריטריוני עובר

- [ ] כל פסילה (❌) מנומקת מול ערך datasheet בפועל — לא מול snippet או סיכום דיסטריביוטר
- [ ] מועמד שה־snippet שלו נראה גבולי / חלקי / חסר פרמטר עדיין נפתח לו datasheet (לא נפסל מראש)
- [ ] אם מאגר המועמדים היה גדול ובוצע סינון־snippet מקדים — כל מועמד שנזרק לפני datasheet מתועד ביומן הכיסוי כ־snippet-filtered (not datasheet-verified)
- [ ] כשמאגר המועמדים קטן — נפתח datasheet לכל מועמד (אין סינון־snippet כלל)
- [ ] הצהרת הכיסוי מבחינה בין "נבדק מול datasheet ונפסל" ל"נזרק בסינון snippet"

## קריטריוני נכשל

- פסל רכיב וכתב את הערך הכושל מתוך snippet בלי לפתוח datasheet
- זרק מועמדים לפני datasheet בלי לתעד זאת כ־snippet-filtered ביומן הכיסוי
- הציג יומן כיסוי שבו "נפסל" ו"נזרק בסינון" נראים זהים

## מה לשמור

קובץ ה־Excel (בדגש על גיליון יומן הכיסוי), הטקסט המלא של התשובה.
